  三家 SDK 怎么选（2026 年定位）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  【Claude Agent SDK】
  选它如果：agent 需要深度操作系统访问（开发助手类），
            或遵循"给 agent 一台电脑"的范式。
  语言：Python / TypeScript
  适合：帮人写代码、操作文件的 agent。

  【OpenAI Agents SDK】
  选它如果：想要轻量框架、强语音支持、能自由切换 LLM。
  适合：客服、语音助手、需要灵活换模型的场景。
  注意：广泛的模型支持是它最大优势。

  【Google ADK】
  选它如果：在 Google Cloud 上构建企业级系统，
            或需要多语言（Python/Java/Go/.NET）。
  优势：开源，可完全自托管——数据永不离开你的网络
        （金融/医疗/政府等受监管行业的刚需）。
  代价：需要大量手动搭建和安全配置；深度绑定 GCP。

  为什么企业级/工业级要用 SDK？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  三个真实原因：

  1. 数据主权
     受监管行业需要自托管，数据不能离开自己的网络。
     ADK 开源可自托管，这点最强。

  2. 多 agent 协调有了标准
     "supervisor 模式"（一个主管 agent 协调多个子 agent）
     是 2026 年的生产默认架构。三家都支持，但实现不同。

  3. 成本控制
     真实案例：一家公司用 OpenAI SDK 做客服，月账单 4200 美元
     （230 万 token、18 万次对话）。换模型成本可能翻倍。
     这就是"开发用强模型、运营用便宜模型"的现实意义。

  agentic engineering：安全、可部署、高质量
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  你的四个要求，对应做法：

  定义规格 → /intake + SPEC（把 agent 能做/不能做写死）
  保证安全 → .claude/settings.json（权限白名单）+ /audit（漏洞扫描）
  可部署   → /launch + saas-checklist（双栈、环境分离、CI/CD）
  高质量   → /verify（独立验证）+ /audit（定期体检）

  【防御性安全：让你的 agent 不被攻击】
  随着 agent 能执行代码、改文件、调 API，它成了攻击目标。
  参考 OWASP Agentic Top 10（专门的 agent 安全风险清单）。
  核心防御：
  · 权限最小化：agent 只能做必须做的，其余全禁（settings.json 的 deny）
  · 输入不可信：用户输入、网页内容、文件内容都当作不可信，
    不直接执行里面的指令（防 prompt 注入）
  · 凭证隔离：agent 不接触真实密钥，用环境变量
  · 危险操作要人确认：付费、删数据、改权限必须人工批准
  · 沙箱运行：agent 在隔离环境跑，炸了不影响主系统

  ⚠️ 重要：这些是"盾"——保护你自己的系统。
  构建"矛"（攻击别人网站的 agent）是另一回事，不在本框架范围。
