适用场景：项目代码已经存在（人写的、AI 写的、混合的都行），现在要把它纳入
本体系，以便测试、修复、改进、降本。本命令替代新项目流程中的 /intake 和
/init-harness。**本命令不修改任何业务代码。**

## 步骤

1. 环境盘点：
   - **【目录锁定·最高优先级】** 运行 pwd，把完整路径报告给用户。除非用户已明确
     指定，否则就在当前目录接管，**绝对禁止 cd 到其他目录、禁止因为别处存在
     同名项目（如某个 research_agent 文件夹）就切换过去**。当前目录不像项目就
     停下来问用户，不要自己去别处找。
   - 如果还不是 git 仓库：git init 并提交一次"接管前快照"。
   - 已是 git 仓库：打标签 git tag adopt-baseline，留下永远可回去的锚点。

2. 通读代码库，生成 docs/SPEC.md ——注意这是**现状说明**，不是理想说明：
   - 这个 Agent 解决什么问题，输入是什么、输出是什么
   - 所有 LLM 调用点清单：位置 / 用途 / 当前模型 / 大致 token 量
   - 外部依赖（API、数据源、凭证从哪来）
   - 文档开头标注：「本文档由代码反推生成于 <日期>，描述现状而非设计意图」

3. 从代码反推生成 feature_list.json（条目结构与 /init-harness 相同）：
   - 把代码中实际实现的每个功能展开成条目，steps 必须具体到照着操作
     就能判断通过与否
   - **全部条目初始 passes: false** ——铁律：没经过本体系端到端验证的
     功能一律视为未验证。代码存在不等于功能可用。
   - 明显该有但代码里缺失的部分（错误处理、空输入、超时、边界），
     也列为条目，category 标 missing

4. 生成或补全 init.sh（一条命令启动整个项目），亲自运行一次确认它能工作。

5. 盘点模型配置：如果模型名散落在代码各处，生成 config.yaml 罗列现状，
   并把「模型配置集中化改造」登记为 feature_list 的一个条目——
   **不要现在就动手改**，留给之后的 /work 会话。

6. 创建 claude-progress.txt，写入首条记录：接管日期、功能总数、已知疑点。

7. 把功能清单分块展示给用户，请用户补充遗漏、剔除反推错误的条目。
   这是唯一一次允许人工直接调整清单的机会，之后一律走 /change。

8. 用户确认后 git commit（提交信息：adopt: 接管建档），然后汇报：
   「接管完成，共 N 条功能待验证。下一步：开新对话运行 /verify 做首次体检。」

## 铁律
- 不修改任何业务代码，只生成文档、配置与清单。
- 反推不确定的地方明确标注"待确认"并向用户提问，不要脑补。
- 接管后的第一个动作永远是 /verify——先体检，后动刀。
