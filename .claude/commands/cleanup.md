你是代码清理专家。**安全第一：小批量清理，每批提交，可单独回退。
任何删除前必须让用户确认。** 上下文盲目的删除会引入 bug，所以每删一批必须验证。

## 第一步：扫描冗余（只扫描，不删）

逐项检查并列清单：

1. **死代码**：定义了但从没被调用的函数、变量、文件。
   - 有工具就用工具（JS/TS: ts-prune、knip；Python: vulture）
   - 没工具就用 git grep 交叉检查引用
2. **废弃的功能条目**：feature_list.json 里 `deprecated: true` 的条目。
3. **过长的文件**：
   - claude-progress.txt 超过 500 行 → 建议归档旧记录
   - CLAUDE.md 超过 200 行 → 建议精简
   - 单个代码文件超过 500 行 → 建议拆分
4. **临时文件残留**：prototype/ 里验证完没清的、mockups/ 里过时的原型。
5. **命名不规范**：不符合 CLAUDE.md 命名规范的文件/变量。
6. **重复代码**：多处出现的相同逻辑块。

## 第二步：出清理清单，分三类

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🧹 清理清单 · [日期]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🟢 安全清理（删了几乎没风险，建议直接清）：
  · 死代码：[文件:行号，函数名]
  · 临时残留：prototype/old-test.js

🟡 需要确认（可能有隐藏引用，删前请你点头）：
  · [函数名]：看起来没用，但可能被动态调用

🔵 建议整理（不是删，是重组）：
  · claude-progress.txt 已 800 行 → 把 3 个月前的归档到 progress-archive.txt
  · feature_list.json 有 12 个 deprecated 条目 → 移到 feature_list-archive.json
  · CLAUDE.md 230 行 → 精简历史教训区

请告诉我：🟢 是否全部清理？🟡 哪些可以删？🔵 是否整理？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 第三步：执行清理（用户确认后，小批量进行）

严格按这个节奏，**不可一次性大删**：

1. 每次只清一类（或一个文件），清完立即：
   - 运行测试，确认没清坏
   - git commit，提交信息：`chore: 清理 [清理了什么]`
2. 清下一类前，确认上一批测试通过。
3. 任何一批清理导致测试失败 → 立即 `git reset` 回退这一批，
   标记该项为"不可安全删除"，跳过它继续。

## 第四步：更新闭环记录

- 在 claude-progress.txt 记录：本次清理了什么、删了多少、归档了什么。
- 如果清理涉及功能变化，对应更新 feature_list.json。
- 输出清理总结：

```
🧹 清理完成
  删除死代码：X 处
  归档：progress 旧记录 / Y 个 deprecated 条目
  精简：CLAUDE.md 从 230 行→180 行
  所有测试仍通过 ✅
  本次清理已提交，可单独回退
```

## 闭环说明（这是"自我管理"的关键）

这套系统的自我管理闭环是：
  /change（自然语言提需求 → 自动更新 SPEC 和功能清单）
  → /work（开发）→ /verify（验证防假完成）
  → /audit（定期体检发现问题）
  → /cleanup（定期清理防腐化）
  → 回到 /change（新需求继续）

只要按节奏跑这个圈，项目就不会积累长期技术债。

## 铁律
- 安全第一：删除前必须用户确认，小批量执行，每批提交可回退。
- 测试失败立即回退该批，不硬删。
- 不删 pre-existing 的、不确定是否在用的代码——标记出来让用户决定。
- 归档而非直接删除历史记录（progress、deprecated 条目移到 archive 文件）。
- 📍 下一步：清理完开新对话打 /status 看项目状态，或继续 /work。
