# /change — 需求变更（中途修改的唯一合法通道）

用户想新增、修改或砍掉功能时运行本命令。**本命令不实现任何代码。**

## 流程

1. 读取 `docs/SPEC.md` 与 `feature_list.json`。

2. 让用户描述变更。逐条澄清到可测试的程度（标准与 /intake 相同：
   出现"等等"、"大概"必须追问）。

3. 生成《变更影响清单》展示给用户：
   - **新增**：将追加哪些新功能条目（分配新 ID，passes 为 false）
   - **废弃**：哪些现有条目将标记 `"deprecated": true` 并附原因——绝不删除条目
   - **受影响**：哪些已经 passing 的功能因本次变更需要重新验证
     （它们的 passes 将改回 false）
   - **规格修改**：SPEC.md 的哪些章节将如何调整
   - **清理工作**：废弃功能若已有对应代码，追加一条"移除 xx 相关代码"的
     新功能条目，让后续 /work 会话完成清理——废代码留着就是技术债

4. 用户**逐条确认**后才落盘：
   - 更新 docs/SPEC.md，并在文末"变更记录"区追加：日期、变更内容、原因
   - 按清单修改 feature_list.json

5. `git commit`，提交信息格式：`change: <一句话摘要>`
   （/verify 审计篡改时，以 change: 开头的提交属于合法变更。）

6. 在 claude-progress.txt 记录本次变更与影响范围。

7. 提示用户：受影响的功能已回到 failing，后续 /work 会自然处理；
   如果变更较大，建议先跑一次 /verify 确认当前基线，再继续开发。

## 铁律
- 不删除任何功能条目，只追加新条目或标记 deprecated——保留完整审计轨迹。
- 不改写已有条目的 description 和 steps。
- 未经用户逐条确认，不得修改任何文件。
- 本命令是除 passes 字段外修改 feature_list.json 的唯一合法路径。
- 本命令结束时必须在最后输出：
「📍 下一步：开新对话，打 /work 实现新增的 N 个功能。」
