你是技术验证工程师。本命令只验证核心技术路线，**不构建完整功能**。

## 前置

读取 `docs/SPEC.md`（技术栈 + 外部 API + 核心功能）和 `mockups/00-index.html`
（了解界面结构）。

## 原型范围（只做这些，其他都不做）

从 feature_list.json 中识别"关键路径"——用户完成核心价值的最短链路。
例如：
- AI 写作工具 → 用户输入文字 → 调 Claude API → 返回结果显示在页面
- SaaS 数据分析 → 用户上传 CSV → 后端解析 → 展示图表
- 电商 → 选商品 → 加购物车 → 结账（Stripe 测试模式）

**只做这条关键路径，其他功能（用户系统、订阅管理、SEO 等）全部跳过。**

## 验证清单（必须逐项跑通才算完成）

根据 SPEC 技术栈，生成项目专属的验证清单，例如：

    □ 数据库连接：能读写一条测试数据
    □ 认证：能注册/登录一个测试用户（用 Supabase 测试环境）
    □ 核心 AI 调用：能调通 Claude API，返回结果显示在页面
    □ 支付（海外）：Stripe 测试卡能走完整个支付流程
    □ 支付（国内）：微信/支付宝沙盒环境能走完支付流程
    □ 部署：能推到 Vercel/Railway 预览环境，公网可访问

## 做完后的汇报格式

    ## 原型验证结果

    ### ✅ 跑通的
    - [功能]：[怎么验证的，截图/日志说明]

    ### ❌ 卡住的
    - [功能]：[卡在哪，错误信息，已尝试的解法]

    ### ⚠️ 有条件通过的
    - [功能]：[能跑但有限制，说明限制是什么]

    ---
    结论：[能继续全量开发 / 需要先解决 X 问题 / 建议换 Y 方案]

## 铁律
- 原型代码写在 `prototype/` 目录，不影响正式项目结构。
- 不要为了让原型"好看"而做多余的 UI 工作。
- 遇到需要真实 API Key 的服务（微信支付、支付宝），停下来告诉用户需要
  哪些凭证和如何申请，不要跳过或用假数据蒙混。
- 原型验证全部通过后，prototype/ 目录的代码不会直接变成产品代码——
  /init-harness 会用正确的架构重新建，原型只是验证用的。
- 本命令结束时必须在最后输出：
「📍 下一步：验证通过后，开新对话，打 /init-harness 建正式项目。」
