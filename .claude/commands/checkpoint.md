当一个会话跑偏、上下文混乱、或需要中途停止时，运行本命令安全着陆，
把状态完整地留给下一个会话。**收到本命令后立即停止一切实现工作。**

## 流程

1. 评估工作区：运行 `git status` 和 `git diff --stat`，向用户汇报当前改动规模。

2. 对未提交的改动三选一（拿不准就问用户）：
   - 改动完整且测试通过 → 正常提交，提交信息说明做到了哪一步。
   - 改动是半成品或未经验证 → `git stash save "checkpoint: <说明>"`，
     并在 progress 中写明 stash 内容与恢复命令；
     如果改动已经把代码搞乱且无保留价值，询问用户后 `git checkout .` 丢弃。
   - 如果用户说"回到上一个好的版本" → 运行 `git log --oneline -5` 展示最近提交，
     让用户选回退到哪个，然后 `git reset --hard <commit>` 回退。
     **回退前必须告诉用户"这会丢失该提交之后的所有改动，确认吗？"**

3. 在 claude-progress.txt 追加一条完整的交接记录：
   - 本会话的原定目标
   - 实际完成到哪一步
   - 为什么中断（卡点的具体描述，包括试过哪些方法、为什么失败）
   - 下一个会话应该从哪里继续、用什么思路
   - 需要避开的坑

4. 如果跑偏的原因可以提炼为规则（例如"在 X 场景下不要用 Y 方法"），
   追加到 CLAUDE.md 的"历史教训"区。

5. 最后告诉用户：
   "状态已保存。请关闭本会话，开一个全新会话继续——
   不要在已经混乱的上下文里恋战，重置比修补便宜。"

## 铁律
- 绝不在收尾时"顺手"完成剩余工作。
- 交接记录必须详细到：一个对项目零记忆的新会话读完就能接手。
- 本命令结束时必须在最后输出：
「📍 现场已保存。开新对话，打 /status 查看状态，然后继续。」
