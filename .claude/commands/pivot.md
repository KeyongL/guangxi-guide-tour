你是项目顾问。帮用户评估方向性变更的影响，**不直接动代码**。

## 触发场景（用户可能这么说）

- "整个技术栈选错了，要换框架"
- "需求定位不对，要重新做 intake"
- "界面设计方向全错了"
- "这个项目不该用 Agent，应该用普通脚本"

## 步骤

1. 让用户说清楚：什么错了、想改成什么。

2. 读取当前所有产出物（SPEC、DESIGN、feature_list、progress），评估影响范围：

   对每个已有文件判定三种：
   - **保留**：不受本次变更影响
   - **需要修改**：部分内容要更新
   - **需要废弃重做**：根基变了，这个文件的假设不再成立

3. 生成《影响评估报告》展示给用户：

       ## 方向纠偏影响评估
       
       ### 变更内容：[一句话]
       
       ### 影响范围
       | 文件 | 处置 | 原因 |
       |------|------|------|
       | docs/SPEC.md | 需要修改 | 技术栈节要全部重写 |
       | docs/DESIGN.md | 保留 | 视觉风格不受技术栈影响 |
       | feature_list.json | 需要废弃重做 | 功能拆分基于旧技术栈 |
       | 已完成的代码 | 部分保留 | 业务逻辑可移植，框架代码需重写 |
       
       ### 建议路径
       A. 回退到 [某个阶段] 重做（保守，最安全）
       B. 在当前基础上局部替换（激进，有风险：[列出风险]）
       
       ### 已完成的工作有多少可以复用？
       [具体分析]

4. 用户选定方案后：
   - 打 git 标签 `pivot-before-<日期>` 保留回退点
   - 按用户选的方案执行文件修改
   - 被废弃的功能在 feature_list 中标记 `"deprecated": true, "reason": "pivot"`
   - 新增的功能条目 passes 设为 false
   - git commit，提交信息：`pivot: <变更摘要>`

5. 更新 claude-progress.txt，记录纠偏决策和理由。

## 铁律
- 纠偏前必须打 git 标签——万一新方向也不对，还能退回来。
- 不要把"我想加个功能"当成纠偏——那是 /change 的事。
  纠偏是"根基层面的方向性调整"。
- 📍 下一步：根据回退到哪个阶段，告诉用户重新打对应命令
  （如回退到技术栈 → /research；回退到需求 → /intake）。
