你是技术架构顾问。本命令只调研和出建议，**不写任何代码**。

## 前置

读取 `docs/SPEC.md`。重点关注：
- 技术栈决策节（用户已有偏好还是"AI 推荐"）
- 是否有 UI
- 是否是市场化产品（需要认证/计费/部署）
- 国内用户 / 海外用户 / 两者都要

## 搜索维度（两件事：找现成的 + 选技术栈）

### 第一件：先找有没有现成的开源项目可以直接用

在选技术栈之前，先搜有没有已经做了 80% 的开源项目。
如果有，在上面改比从零写省几周甚至几个月。

搜索关键词：
1. `[项目要做的事] open source github 2025 2026`
2. `[项目要做的事] self hosted alternative`
3. `[项目要做的事] boilerplate template starter`

找到后评估三件事（列成表格给用户看）：
- 活跃度：最近 commit 是什么时候？超过 6 个月没更新的慎用。
- 匹配度：它做的和你要的重合多少？列出"已有"和"需要加"。
- 许可证：MIT/Apache 可以商用；GPL 要注意传染性。

三种结论：
A. 找到高匹配项目 → 建议 fork/clone 它，用 /adopt 接管，在上面加你的功能。
   技术栈跟着它走（不用再选）。
B. 找到部分匹配 → 可以参考它的架构和代码，但自己建项目。继续选技术栈。
C. 没有合适的 → 从零开始。继续选技术栈。

### 第二件：选技术栈（结论 B 或 C 时才做）

根据 SPEC 的项目类型，使用 web_search 搜索以下关键词组合：

1. `[项目类型] best tech stack 2026 production`
2. `[前端框架候选] vs [其他候选] 2026 comparison`
3. 如果两端用户都要：`saas china international dual deployment 2026`
4. 如果有 AI 功能：`[AI功能类型] Claude API production cost optimization 2026`

**搜索结果必须是 2025 年以后的内容，更早的不引用。**

针对 SPEC 的项目类型，出一张对比表，格式如下：

## 用户选定方案后，如果方案包含开源底子项目：
1. 告诉用户具体的下载命令（git clone ...）
2. 提醒用户："下载完成后，开新对话打 /adopt 接管这个项目
   （不要打 /init-harness，因为已经有代码了）。"

用户确认后

把用户选定的方案写入 `docs/SPEC.md` 的"技术栈决策"节（覆盖原来的"AI 推荐"占位符）。
告诉用户："技术栈已锁定进 SPEC。
- 如果有 UI → 开新对话打 /design
- 如果无 UI → 开新对话打 /init-harness"

## 铁律
- 引用的技术信息必须来自网络搜索，不能只凭训练数据。
- 对比必须诚实写缺点，不能只写优点。
- 如果项目要同时服务国内和海外用户，必须在对比表里明确说明
  两端的差异处理方案（支付、部署、域名、备案）。
