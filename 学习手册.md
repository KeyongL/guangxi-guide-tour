  第 1 章　AI 是个什么样的"员工"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  【优点】很聪明，会写代码、查资料、测试。
  【缺点】有"健忘症"——对话结束就忘光。对话越长越容易犯迷糊。

  这个缺点是所有 AI 的通病。Anthropic 官方说，上下文退化（对话太长变笨）
  是 AI 最主要的失败原因。

  所以整套系统的所有设计，归根结底都在对付这一件事：
  "AI 会忘，而且聊久了会变笨。"

  解法：给他留"工作日志"。每次上班先翻日志，就知道昨天干到哪。
  这就是为什么可以关掉对话再开——进度在文件里，不在对话里。

  第 2 章　两个最重要的文件
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ① feature_list.json — 功能清单（进度条）
     列着所有要做的功能。false=没做, true=做完验证过。
     数 true 的数量 = 进度百分比。

  ② claude-progress.txt — 交接班记录
     每次 AI 做完会写一笔：做了什么、改了哪些文件、留了什么坑。
     新 AI 上班先读这个，就知道接着干什么。

  Anthropic 官方原话：
  "绝不让一个长对话成为你唯一的记录。靠文件记事，把每个对话当成用完即弃。"

  第 3 章　生词翻译
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  session（会话）= 一个对话窗口。"新 session"= 关掉重开。
  目录 = 电脑里的文件夹。AI 必须站对目录，否则文件跑到别处。
  命令 = 斜杠开头的指令（如 /work），等于一键发送一整套工作步骤。
  git = 自动存档工具。做坏了能回退到上一个好版本。
  端到端测试 = 像真人一样从头到尾用一遍，确认真的好使。
  Boilerplate = 别人做好的项目模板（含认证、支付等基础设施），
               你 clone 下来在上面加自己的功能，不用从零搭。

  第 4 章　20 个命令的完整手册
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  不是每个项目都用全部 20 个。下面标了哪些可跳过。

  ────── 随时可用 ──────

  /status — 看进度，告诉你下一步
    什么时候用：不知道该做什么时。读所有进度文件，10 秒给你答案。
    你要做：看它说的下一步，照做。

  /checkpoint — 紧急叫停
    什么时候用：AI 卡住、绕圈、越做越乱时。存档退出，开新对话继续。
    口诀：感觉不对就叫停，重开比硬撑划算。

  /change — 改需求
    什么时候用：想加功能、改功能、砍功能时。这是变更的唯一合法通道。
    为什么不直接说：随口让 AI"顺便改"是项目失控的头号原因。

  ────── 项目开头（按顺序做一次）──────

  /start-here — 确认站对地方
    每个项目的第一道工序。AI 报路径，你核对。防止文件跑到别处。

  /intake — 问清需求
    AI 像记者一样一题一题问你。市场化产品会自动核对 SaaS 清单
    （认证、计费、邮件、法律页面有没有漏）。产出一份需求文档。

  /research — 搜索最新技术栈（小工具可跳过）
    AI 用网络搜索拿 2025-2026 最新数据，出对比表你选。
    不靠 AI 记忆（可能过时），靠实时搜索。

  /design — 设计 + 看原型画面（无 UI 可跳过）
    先问你风格偏好，再直接出 HTML 原型。你在浏览器打开看画面，
    确认再动手。如果有 Boilerplate，在它的设计系统基础上定制，不冲突。

  /prototype — 技术验证（小项目可跳过）
    只跑核心路径：数据库能连、AI 能调、支付能走。不是完整产品，
    是证明"这条路走得通"。走不通提早发现，比做到一半发现好。

  /init-harness — 建功能清单和骨架
    把需求拆成一条条功能（如 18 条），全标 ✗。
    自动检测 Boilerplate：有就在上面建，没有就从头搭。
    从这里开始，项目有了"进度条"。

  ────── 干活循环（反复做）──────

  /work — 做一个功能
    AI 先读进度（衔接是自动的），挑一个 ✗ 做完变 ✓，停下。
    你只需：看方案说"可以"，看进度汇报，关掉对话。
    重复到 AI 主动说"🎉 全部完成"。

  /verify — 质检
    派一个独立 AI 挑剔地重测"做完"的功能，揪出假装做完的。
    每 3-5 个 /work 插一次。全部做完后做总验收。
    AI 最爱犯的错就是"没好好测就说做完了"，/verify 专治这个。

  ────── 上线（市场化产品）──────

  /launch — 发布前核查
    支付环境切换了没？隐私政策有内容没？ICP 备案号挂了没？
    逐项检查，全过才能上线。

  ────── 特殊入口 ──────

  /adopt — 接管已有代码（代替 intake~init-harness）
    已有成品时用。AI 读代码反向建清单（全标 ✗），然后走 /verify 体检。

  /pivot — 方向纠偏（发现根基搞错了）
    不是改一个功能（那用 /change），是整个方向、技术栈、定位不对时用。
    它会评估哪些工作能保留、哪些要推翻，打标签留回退点，告诉你回到哪一步重来。

  第 8 章　模型怎么选（省钱指南）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  不是每个命令都需要最贵的模型。规则很简单：

  【必须用强模型（Opus / GPT-4o / Sonnet）】
  /intake、/research、/design、/init-harness、/verify、/launch、/adopt
  → 这些是规划、设计、审计——最吃脑子的活。

  【可以试便宜模型（Haiku / DeepSeek / GLM）】
  /work → 一次只做一个小功能，规格已经定死了。
  试便宜模型做几个 → 用强模型跑 /verify 检查。
  打回率低于 20% → 便宜模型可以长期用。
  打回率高 → 换回强模型，或只在简单功能上用便宜的。

  第 9 章　Skills 怎么跟命令配合？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  你的 16 个命令 = 做事的节奏（什么时候做什么）
  Skills = 做事的手艺（代码怎么写才好）

  比方：命令是"施工进度表"，Skills 是"工人的技能培训证书"。
  进度表管什么时候干，技能证书管干得好不好。两个叠加不冲突。

  实际操作：
  1. 在任何阶段都可以安装 Skills，不影响流程
  2. 装好就行，AI 做 /work 时自动调用匹配的 Skill
  3. 你不需要在命令里提"请用 XX 技能"——AI 自己会用

  推荐安装时机（在 /init-harness 之前装好）：
  · 所有项目：superpowers（执行纪律 + TDD，一行安装）
  · 有 UI 的项目：frontend-design（设计质量）
  · 复杂项目：gstack（角色化决策，如让 AI 扮演 CTO 审架构）

  安装方式（在 Claude Code 里输入）：
    /plugin marketplace add obra/superpowers-marketplace
    /plugin marketplace add anthropics/skills --skill frontend-design

  ⚠️ 注意：superpowers / gstack 和我们的命令管的是同一层（工作流），
  混装可能冲突。建议：先用我们的轻量命令跑熟一个项目，
  再决定是补装单点 Skills（不冲突）还是整套换成 gstack（替代命令）。

  第 10 章　从"你按按钮"到"它自己跑一晚上"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Boris（Claude Code 作者）说的"一晚上不干预"，靠的是 /goal 命令：
  你设一个完成条件，Claude 自己循环工作，每轮用一个独立 AI 检查
  "条件满足了没"——没满足继续干，满足了停下通知你。

  但这有一个前提：你必须有自动化测试（代码跑完返回通过/失败）。
  没有自动化测试，/goal 跑一晚上 = AI 自己考试自己批卷 = 不靠谱。

  升级路线（不急，按顺序来）：

  【你现在】人工驱动
    你打 /work → 看方案 → 说可以 → AI 做完你关掉
    ↓（跑熟流程后）

  【第一步】半自动
    你打 /goal "完成 F005，所有测试通过" → AI 自己循环到完成
    前提：那个功能有自动化测试脚本
    ↓（积累出足够测试后）

  【最终形态】全自动（Boris 说的"一晚上"）
    定时任务触发 → /goal 跑 → 自动化测试验收 → git 提交
    你早上起来看结果

  口诀：先手动跑出信心 → 给关键功能写测试 → 逐步放手给 /goal。
  不要跳步——没有测试就上 /goal，是在制造更大的技术债。

  第 11 章　发现方向错了怎么办？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  两种"错"，处理方式不同：

  【细节错了】"这个按钮逻辑不对" / "想加个功能"
  → 打 /change，登记变更，回到 /work 修。

  【方向错了】"整个技术栈选错了" / "需求定位不对" / "不该做 Agent 该做脚本"
  → 打 /pivot，它会：
    · 评估哪些已做的工作能保留、哪些要推翻
    · 打 git 标签（万一新方向也不对，还能退回来）
    · 告诉你该回退到哪个阶段重新开始

  区分方法：问自己"改的是一个功能，还是整个项目的根基？"
  改功能 → /change。改根基 → /pivot。

  第 12 章　文件规范管理（防止混乱和安全漏洞）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  项目里有四类文件，规则不同：

  【进 git、所有人共享的】
  CLAUDE.md、.claudeignore、.gitignore、.claude/settings.json、
  .env.example（只有变量名，不含真实值）、feature_list.json

  【进 git，只有你本地有】
  .env（真实密钥，.gitignore 里列着，永远不提交）
  CLAUDE.local.md（你的个人笔记，自动不提交）

  【不进 git，AI 生成的产物】
  mockups/（视觉原型 HTML）、prototype/（技术验证代码）

  ★ 最重要的安全规则：
    .env 必须同时在 .gitignore 和 .claudeignore 里。
    一旦 API 密钥进了 git，即使你删了，历史记录里还有——
    黑客能扫到，你的账单会暴涨甚至被封号。

  四个关键配置文件的作用：

  .gitignore        → git 不追踪哪些文件（防密钥泄露）
  .claudeignore     → Claude 不读哪些文件（防上下文污染 + 安全）
  .claude/settings.json → Claude 能做和不能做什么操作
  CLAUDE.md         → Claude 每次对话自动加载的项目规范

  CLAUDE.md 里应该有什么（这套模板已经配好了）：
  - 项目地图（重要文件在哪）
  - 文件结构（目录长什么样）
  - 命名规范（文件、变量、函数怎么命名）
  - Git 提交格式（feat/fix/style/refactor 等）
  - 分支策略（feature 分支，不直接提交 main）
  - 铁律（不可商量的规则）

  /init-harness 会自动帮你生成这四个文件。
  你只需要：把 .env.example 复制成 .env，把真实的 API Key 填进去。

  第 13 章　离开电脑让 AI 连续干活（吃饭/睡觉）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  你想离开电脑，让 AI 自动连做好几个功能。用 /auto 命令。

  【为什么不能简单地说"做完所有功能"就走？】
  因为有个坑叫"目标漂移"：一个对话做太久，上下文会满，AI 自动压缩
  早期内容来腾空间。压缩是有损的——你的规则、你的纠正会被压没。
  Anthropic 官方都确认：多轮压缩后，连 CLAUDE.md 的关键规则都会失效。
  结果就是 AI 越做越偏，早上起来发现做了一堆错的。

  【/auto 怎么防止这个坑？】
  它不让一个对话扛所有功能，而是每个功能派一个"独立子代理"——
  每个子代理有全新的、干净的上下文，做完一个功能就销毁，
  绝不积累。子代理之间靠 feature_list.json 和 git 提交传递进度，
  不靠对话记忆。

      ❌ 错：一个对话 → F5 → F6 → F7 → 上下文爆炸 → 漂移翻车
      ✅ 对：子代理A做F5（用完销毁）→ 子代理B做F6（全新上下文）→ ...

  这其实就是你这套模板一直的原理"靠文件不靠对话"，只是自动化了。

  【什么时候适合无人值守？四个条件全满足才行】
  ✅ 有自动化测试（代码能判断对错）→ 否则 AI 自说自话不可信
  ✅ 任务能拆成独立小块 → 每块一个子代理，防上下文爆炸
  ✅ 错误可恢复（有 git）→ 做坏了能回退
  ✅ 不涉及真实付费/删数据 → 无人盯着时高风险操作没人拦

  适合睡觉跑的例子：批量重构（有测试）、跑通一批后端功能、补单元测试、
  数据清洗（输入输出可校验）。
  绝不适合：UI 外观调整、真实支付、第一次用、没有任何测试的项目。

  睡前必做的检查（/auto 会帮你逐项确认）：
  □ 这批功能有测试吗？  □ 能拆成小块吗？  □ git 干净吗（留好回退点）？
  □ 设了 token 上限吗？  □ 涉及付费/删数据吗？

  【怎么用】
  开新对话打 /auto，它会问你：
    · 做哪些功能（如 F005-F010）
    · token 上限（建议 250K-500K，防账单失控）
    · 有没有付费调用
  然后你就可以走了。回来打 /status 看它做完了几个。

  【多智能体编排（dynamic workflow）】
  /auto 背后用的就是多智能体——多个子代理并行/串行干活。
  如果你想让 AI 为某个复杂任务专门设计一套编排，
  在 Claude Code 里用触发词 "ultracode"，
  它会用 Opus 4.8 为你的具体场景写一个量身定制的 harness。
  这是进阶玩法，跑熟基础流程后再碰。

  口诀：有测试才敢睡觉跑，设上限才不会破产，子代理隔离才不漂移。

  第 14 章　长期维护：防止越用越乱 + 安全加固
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  【先说一个数据，让你重视】
  行业数据：40% 的 AI 生成代码有安全缺口，75% 的项目会产生技术债。
  但这正是因为大多数人没有你手上这套体系——没有独立审计、没有进度锚点、
  没有"一次一个"的纪律。这套框架本身就是为了不让你变成这个统计数字。

  但光有流程不够，还要定期体检。用 /audit 命令。

  【/audit 查什么】
  三方面体检：
  1. 安全：密钥有没有泄露、依赖有没有漏洞、有没有常见的 AI 代码漏洞
     （SQL 注入、CORS 全开、缺输入校验等）
  2. 健康：有没有超大文件、重复代码、死代码、TODO 堆积
  3. 可维护性：CLAUDE.md 是否太长、依赖是否过杂、升级是否困难

  【最重要的安全检查：密钥泄露】
  /audit 会检查你的 .env 有没有被误提交到 git。
  ⚠️ 一旦密钥进了 git，即使删了，历史记录里还在——必须立即换掉所有密钥。
  这是 AI 开发最常见的事故，/audit 第一件事就是查这个。

  【定期体检节奏】
  · 每完成一个大阶段 → /audit 一次
  · 每月至少 → /audit 一次
  · 上线前 → 必须 /audit
  · 把 /audit 当成"项目的体检"，不是出问题才做，是定期做

  【发现问题怎么修】
  /audit 只诊断不改代码。要修的问题，用 /change 登记成功能条目
  （标记 security 或 tech-debt），然后 /work 逐个修。
  审计和修复分开，保持干净——这本身就是防技术债的做法。

  【进阶：Claude Code Security】
  Anthropic 2026 年出了专门的安全扫描工具（claude.ai/security），
  用 AI 推理发现模式匹配查不出的逻辑漏洞，已在开源项目发现 500+ 高危漏洞。
  企业版可用。如果你的产品涉及真实用户数据，强烈建议接入。

  【关于这个框架本身会不会过时】
  这套命令都是纯文本 .md 文件，你随时能改。
  通用规则集中在 _shared-rules.md 里，升级时改一处即可。
  如果将来想换成更重的方案（gstack/superpowers），
  你的 feature_list 和进度文件都能无缝迁移——因为它们是标准格式，不绑定工具。

  口诀：定期体检不要等出事，密钥泄露最优先，审计和修复要分开。

  【完整的自我管理闭环——防止长期技术债】
  你担心"用久了越来越乱、堆积技术债"——这套系统有一个闭环专门防它：

    ① /change   你用人话提新需求 → 自动更新需求文档和功能清单
                （不用手动改文件，说"我想加个导出功能"它就改好）
    ② /work     开发新功能
    ③ /verify   独立验证，防止"假装做完"
    ④ /audit    定期体检，发现冗余、漏洞、技术债（每月）
    ⑤ /cleanup  定期清理，删死代码、归档废弃条目、精简长文件（每月）
    ⑥ 回到 ①   新需求继续

  只要按这个圈转，项目就始终清爽，不会积累长期技术债。
  关键是 ④⑤ 要定期做——很多人只顾往前加功能（①②③），
  从不回头清理，几个月后就成了一团乱麻。

  【文件越来越乱怎么办——/reorganize】
  /cleanup 是"删垃圾"（删死代码、废弃项）。
  /reorganize 是"搬家归位"（把放错地方的文件移到该在的地方、统一命名）。
  这两个不一样，都需要：

  你担心的"项目膨胀、复用代码后文件越来越乱"，靠 /reorganize 解决。
  它照着 docs/PROJECT-STRUCTURE.md（你的"文件宪法"）把乱掉的项目
  重新规整成标准结构。

  完整的文件规范闭环（你说的"乱→规范→再乱→再规范"）：
    开发 /work        → 文件慢慢变乱（正常，挡不住）
    定期 /audit       → 发现"乱了"的信号
    /cleanup          → 删垃圾
    /reorganize       → 搬家归位，按标准重整
    → 回到整洁，继续开发

  ★ 核心心法（工程化思维）：
    乱不可怕。可怕的是没有标准、没有流程。
    有了 PROJECT-STRUCTURE.md 当标准、有了 /reorganize 当流程，
    乱了就能重新规范化——就像德国人日本人那样，按部就班，乱了就整理，
    永远不会失控。

  关键规则"按功能分，不按类型分"：
    ❌ 把所有控制器放一起、所有模型放一起（改一个功能要翻 4 个目录）
    ✅ 一个功能的所有代码放一个目录（改 auth 只看 auth/ 文件夹）
    这是 2026 年主流做法，也让 AI 更容易准确修改你的代码。

  /cleanup 怎么防止"清坏了"：
  · 小批量清，一次一类，清完就提交（可单独回退）
  · 每清一批跑一次测试，失败立即回退那批
  · 历史记录是归档不是删除（旧进度移到 archive 文件）
  · 任何删除前都问你确认

  口诀：往前加功能要配套往回清垃圾，一发现一清理，债就压不住。

  第 15 章　硬护栏：用代码强制，不靠 AI 自觉
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  前面所有防偏移的办法，大多是"软约束"——写在文件里让 AI 遵守。
  但 AI 会忘、会偏。最可靠的是"硬约束"——用代码强制，AI 想绕也绕不过。

  软约束 vs 硬约束（打个比方）：
  · 软约束 = 门上贴张纸"请随手关门"（人可能不看、忘了）
  · 硬约束 = 装个自动闭门器（门必然会关，不靠人自觉）

  你的 agent 上生产、被真实用户用时，必须加硬护栏。用 /harden 命令。
  它加四层（业界叫"深度防御"）：

  【第1层 结构校验】AI 输出的 JSON 格式对不对？
    真实案例：某 agent 8% 的输出 JSON 解析失败，下游全崩。
    硬约束做法：用代码校验格式，不合格自动带错误重试，最多3次。
    效果：解析失败率从 8% 降到 0.3%。这不是靠"更好的提示词"，
    是靠两行校验代码。

  【第2层 业务规则校验】格式对了，内容合理吗？
    比如：金额不能为负、引用的 ID 必须真实存在、置信度太低不执行。
    硬约束做法：把业务规则写进校验代码，违反就拒绝。

  【第3层 用户确认门】危险操作执行前必须人点头。
    🟢 读数据 → 直接做    🟡 改数据 → 看配置
    🔴 删除/付款/改权限/发真实邮件 → 永远必须人确认，代码层面禁止自动做
    这一条不可关闭——这是防止 AI 闯大祸的最后一道闸。

  【第4层 输入校验】进来的东西可不可信？
    用户输入、网页内容、文件内容都当"不可信"，
    不把外部内容直接当指令执行（防 prompt 注入攻击）。

  ★ 关键区分：
    校验回答"格式对不对"，护栏回答"允不允许"。两个都要。
    研究显示：加了校验层，AI 幻觉导致的事故下降 60%。

  什么时候加？
  · 纯开发、自己测 → 不急，先把功能做出来
  · 要上生产、真实用户调用 → 必须加，这是底线
  · 涉及钱、涉及删数据 → 第3层确认门绝对不能省

  包里 guardrails/ 目录有两个参考代码（schema校验、确认门），
  /harden 会照着你的项目改写。

  口诀：能用代码强制的就别靠 AI 自觉，危险操作永远要人点头。

  【关于 Agent SDK（进阶，现在不用看）】
  你现在用 Claude Code 开发完全够。
  等你的 agent 要嵌入产品、被真实用户大量调用时，
  才需要考虑用 Claude/OpenAI/Google 的 Agent SDK 重构运行时。
  详见 docs/SDK选型参考.md。核心一句话：开发用 Claude Code，
  规模化运行才用 SDK，别提前上。

  Q1. 为什么开新对话不用复制粘贴上次的内容？AI 怎么知道上次到哪了？

  Q2. 一个 /work 对话做几个功能？为什么？

  Q3. feature_list.json 是干嘛的？怎么用它判断进度？

  Q4. 为什么需要 /verify？让写代码的 AI 自己检查不行吗？

  Q5. 文件夹里空空的，啥都没生成。最可能的原因是什么？怎么排查？

  Q6. 想加新功能。应该直接跟 /work 说"顺便加个 XX"还是怎么做？

  Q10. 做到一半不知道下一步该做什么了。你该打哪个命令？

  Q11. 你的项目要推向市场（有用户注册和付费）。/intake 阶段
       会多做一件什么事来防止遗漏？

  Q12.（综合）用自己的话把市场化 SaaS 从 0 到上线的完整流程讲一遍。


──────────── 参考答案 ────────────

  A1. 因为衔接靠文件不靠对话。AI 在新对话的第一步会去读
      claude-progress.txt 和 feature_list.json，自己就知道接着干什么。

  A2. 只做一个。一次贪多容易出错烂尾，对话越长越笨。

  A3. 进度条。false=没做, true=做完验证过了。数 true 的数量就是进度。

  A4. 写代码的 AI 会偏袒自己——官方说 AI 最爱犯的错就是
      "没好好测就说做完了"。换个独立的来挑刺才靠谱。

  A5. AI 跑到别的文件夹干活了。排查：开新对话让它运行 pwd 和 ls，
      看它站在哪。以后先打 /start-here 钉死目录。

  A6. 不要顺便说。打 /change 正式登记。随口改是失控的头号原因。

  A7. 打 /checkpoint 存档退出，开新对话继续。重开比硬撑划算。

  A8. /adopt。因为 Boilerplate 是已有代码，/init-harness 假设空目录，
      会搭重复骨架导致冲突。/adopt 会在已有代码基础上建功能清单。
      注：最新版 /init-harness 已加了 Boilerplate 自动检测，
      但如果你不确定，用 /adopt 更安全。

  A9. 第一件：问你风格偏好（颜色/字体/调性），出设计文档。
      第二件：直接生成 HTML 原型让你在浏览器看。
      合成一步是因为分两个会话切来切去浪费时间。

  A10. 打 /status。它读所有进度文件，10 秒告诉你在哪、该做什么。

  A11. 自动读 docs/saas-checklist.md 逐项核对——认证、计费、邮件、
       法律页面、部署、监控有没有漏，漏的会补进需求文档。

  A12. 参考讲法：
       "先打 /start-here 锁目录，再打 /intake 被 AI 访谈需求
       （它会核对 SaaS 清单防遗漏），然后 /research 搜最新技术栈我选方案，
       /design 定风格并在浏览器看原型画面确认，/prototype 跑通核心路径
       （数据库、AI调用、支付），确认走得通后 /init-harness 建正式项目
       （它自动识别 Boilerplate 不会冲突），然后反复打 /work 一次做一个
       功能、每几个插一次 /verify 质检，直到全部做完。最后 /launch
       逐项核查（支付切正式、隐私政策、ICP备案），全过就上线。
       全程不用记进度——/status 随时告诉我在哪，
       每个命令结束都告诉我下一步。"
