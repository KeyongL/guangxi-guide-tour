# /work — 工作会话（每次只完成一个功能）

你是编码代理。本会话的目标：让 feature_list.json 中**恰好一个**功能从 failing 变为
passing，并把环境留在干净状态（干净状态 = 可以合并进主分支的程度：无明显 bug、
代码有序、下一个会话不需要先收拾烂摊子）。

## 第一步：启动仪式（不可跳过，按顺序执行）

1. **【目录锁定·最高优先级】** 运行 `pwd`，把完整路径报告给用户，例如
   "我当前在 /Users/xxx/项目目录"。除非用户已在本对话明确指定目录，否则
   **就在当前目录工作，绝对禁止 cd 到其他目录、禁止因为别处存在同名项目而切换过去**。
   如果当前目录看起来不像项目目录（缺 feature_list.json 等），停下来问用户，而不是
   自己去别处找。
2. 读 `claude-progress.txt` 的最后 20 行，了解上一个会话做了什么、留下了什么坑。
3. 读 `feature_list.json`，统计当前 passing / failing 数量。
4. 运行 `git log --oneline -10`，查看最近提交。
5. 运行 `./init.sh` 启动环境。
6. 对 2-3 个已 passing 的核心功能做快速冒烟测试。
   **如果发现已损坏的功能：先修复它，再考虑新功能。** 在坏地基上盖楼只会更糟。

## 第二步：选择与计划

7. 从 failing 功能中选优先级最高的**一个**（依赖别人的功能往后排）。只做这一个。
8. 用 3-6 行写出实现方案：要改哪些文件、怎么端到端验证。展示给用户。
   - 方案涉及架构改动、新增依赖、或删除已有代码 → 等用户确认再动手。
   - 普通实现 → 展示后直接继续。

## 第三步：实现与验证

9. 实现该功能。全程遵守 CLAUDE.md 的所有铁律。
10. 像真实用户一样端到端测试：
    - **后端/API 功能**：用真实输入跑入口命令或发 API 请求，确认输出正确。
    - **UI 功能**：如果有浏览器自动化工具（Puppeteer/Playwright），用它截图验证；
      没有的话，启动开发服务器后明确告诉用户"请在浏览器打开 localhost:XXXX，
      确认页面显示正确后回复'确认'"——**UI 功能的最终验收必须有人眼确认**：严格按该功能的 steps 逐步操作并确认每一步。
    有浏览器自动化工具就用它走完整流程；命令行程序就用真实输入跑入口命令。
11. 端到端验证全部通过后，才把该功能的 passes 改为 true。

## 第四步：收尾（干净状态）

12. 运行测试套件，全部通过后 `git add -A && git commit`，
    提交信息格式：`F00X: <做了什么>`。
13. 在 claude-progress.txt 追加：日期、完成的功能 ID、改动的文件、遗留问题。
14. 如果本会话你犯过任何错误（被用户纠正、走过弯路、误改文件），
    把教训提炼成一条规则，追加到 CLAUDE.md 的"历史教训"区。
15. 汇报进度（passing X / 总数 Y），然后停止。
    **不要继续做下一个功能。** 下一个功能属于下一个会话。

## 铁律
- 一个会话一个功能，做完即停。
- 禁止修改 feature_list.json 中除 passes 之外的任何内容。
- 禁止修改与本功能无关的代码。
- 没有端到端验证就标 passing 是不可接受的。
- 卡住超过 3 次同类尝试 → 停下，把卡点写进 progress，建议用户运行 /checkpoint。
- 本命令结束时必须在最后输出以下三种之一：
· 还有功能未完成，且距上次 /verify 不足 3 个：
  「📍 进度：X/Y 完成。下一步：开新对话，打 /work 继续。」
· 还有功能未完成，但已连续完成 3 个以上未经质检：
  「📍 进度：X/Y 完成。⚠️ 已连续 3 个功能未质检，下一步：开新对话，打 /verify。」
· 所有功能完成（feature_list.json 全部 passes:true）：
  「🎉 所有 Y 个功能已完成！下一步：开新对话，打 /verify 做最终验收。」
