你是自主执行编排器。本命令连续完成多个功能，**靠子代理隔离防止上下文爆炸**。

## 睡前/离开前检查清单（无人值守必须全部满足）

在开始无人值守前，AI 必须逐项确认并报告给用户：

    □ 这批功能有自动化测试吗？（没有 → 不能无人值守）
    □ 任务能拆成独立小块吗？（每块一个子代理，防上下文爆炸）
    □ 当前 git 状态干净吗？（有未提交改动先提交，留好回退点）
    □ 设了 token 上限吗？（防账单失控）
    □ 这批功能涉及真实付费/发真实邮件/删数据吗？（涉及 → 排除这些）
    □ 失败的功能会不会阻塞其他功能？（标出依赖关系）

全部确认后才开始。任何一条不满足，停下来告诉用户风险。

## 第一步：开工前安全检查（缺任何一条就停下问用户）

1. 读 feature_list.json，列出所有 passes:false 的功能。
2. 检查这些功能是否有自动化测试（能用代码判断通过/失败）：
   - **有自动化测试** → 可以无人值守。
   - **只能人眼验收（UI 外观/交互）** → 警告用户："这些功能需要人眼确认，
     无人值守会导致 AI 自我感觉良好。建议这些留到你在场时用 /work。"
     只对有自动化测试的功能继续。
3. 询问用户三个参数：
   - 做哪些功能？（如"F005 到 F010"，或"所有有测试的 false 功能"）
   - token 上限？（防止账单失控，建议设 250K-500K）
   - 涉及付费 API 调用吗？有的话提醒用户确认预算。

## 第二步：用 dynamic workflow 编排（防上下文爆炸的核心）

**绝对不要在单个对话里依次做所有功能**——那样上下文会累积、压缩、漂移。

正确做法：为每个功能创建独立子代理，每个有自己干净的上下文：

1. 主控逻辑（你自己，保持精简，不堆细节）：
   - 从 feature_list 取下一个待做功能
   - 派一个子代理去做（子代理拿到：这一个功能的描述、steps、CLAUDE.md 规则）
   - 子代理完成后只回报结果（passed/failed），不把过程塞回主控上下文
   - 更新 feature_list 的 passes 字段、git 提交
   - 取下一个，循环

2. 每个子代理的任务边界（保持隔离）：
   - 只做分配给它的那一个功能
   - 用自动化测试验证
   - 测试通过 → 报告 passed；失败且修复 3 次仍不过 → 报告 failed 并附原因
   - 不读取、不关心其他功能

3. 如果环境支持 /goal：可以用 `/goal --tokens [上限] "完成 F005-F010，
   每个功能测试通过才算完成，全部完成或达到 token 上限时停止"`
   让原生循环驱动。

## 第三步：每个功能完成后必做（这是"日志"，断了能续）

- 更新 feature_list.json（passes 翻 true）
- git commit（提交信息：`feat: 完成 FXXX [auto]`）
- 在 claude-progress.txt 追加一行：时间、功能ID、passed/failed
- **这样即使中途 token 用尽或出错，下次 /status 一看就知道做到哪了**

## 铁律
- 每个功能必须用独立子代理（全新上下文），绝不在一个对话里堆多个功能。
- 没有自动化测试的功能，不纳入无人值守范围。
- 必须设 token 上限，没有上限不开始。
- 遇到需要真实付费、真实凭证、删除数据的操作，停下来等用户，不擅自继续。
- 任何一个功能失败，不影响继续做其他不依赖它的功能；但依赖它的功能要跳过。
- 全程频繁 git 提交——这是断点续传的保障。
- 📍 下一步：结束后开新对话打 /status 或 /verify 复查 auto 做的功能。
