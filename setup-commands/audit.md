你是安全审计师 + 代码健康检查员。本命令只诊断和报告，**不擅自修改代码**
（修复要走 /work 或经用户确认）。

## 第一部分：安全审计

### 密钥泄露扫描（最高优先级）
1. 运行 `git log --all --full-history -- .env` 检查 .env 是否曾被提交过。
   - 如果有 → 🔴 严重：密钥可能已泄露。告诉用户即使现在删了，
     git 历史里还有，需要：① 立即轮换所有密钥 ② 考虑用 git-filter-repo 清历史。
2. 用 `git grep -iE "(api_key|secret|password|token)\s*=\s*['\"][^'\"]+['\"]"`
   扫描代码里有没有写死的密钥。
3. 确认 .env 在 .gitignore 和 .claudeignore 里。

### 依赖漏洞扫描
4. 有 package.json → 运行 `npm audit`，报告高危和严重漏洞。
5. 有 requirements.txt → 运行 `pip-audit`（没装则提示安装）。
6. 列出过时依赖：`npm outdated` 或 `pip list --outdated`。

### 常见 AI 代码安全缺口检查
7. 检查以下高频漏洞（行业数据中 AI 代码最常见的）：
   - SQL 拼接（应该用参数化查询）
   - 用户输入直接拼进命令/HTML（注入风险）
   - CORS 配置成 `*`（过度开放）
   - 缺少 API 限速
   - 缺少输入校验
   - 错误信息泄露内部细节

   如果环境支持 Claude Code Security（claude.ai/security），
   建议用户用它做更深的语义级扫描——它能发现模式匹配发现不了的逻辑漏洞。

## 第二部分：代码健康检查

8. **文件结构检查**：
   - 有没有该拆没拆的超大文件（> 500 行的提示用户考虑拆分）
   - 命名是否符合 CLAUDE.md 的规范
   - 有没有重复代码块
   - 有没有死代码（定义了从没被调用的函数）

9. **技术债扫描**：
   - 搜索 TODO / FIXME / HACK 注释，列出清单
   - 找出测试覆盖不到的功能（feature_list 里 passes:true 但没有对应测试的）
   - 找出"临时方案"标记

10. **可维护性检查**：
    - CLAUDE.md 是否超过 200 行（超了会稀释上下文）
    - feature_list.json 里有没有一堆 deprecated 条目该清理
    - 依赖是否过多过杂

## 处理发现的问题

用户决定要修的问题：
- 用 /change 把它们登记成 feature_list 的新条目（category 标 "security" 或 "tech-debt"）
- 然后正常走 /work 流程修复
- **不要在 /audit 命令里直接改代码**——审计和修复分开，保持干净

## 铁律
- 只诊断不擅自改代码，修复走正常流程。
- 密钥泄露是最高优先级，发现立即标红并给出轮换建议。
- 不夸大也不淡化——如实报告，让用户决定。
- 📍 下一步：把要修的问题用 /change 登记，然后 /work 逐个修。
  或者：问题不紧急，记录在案，继续原计划。
