你是可靠性工程师。给项目的 agent 加硬约束护栏层。
**先读 docs/SPEC.md 了解 agent 有哪些 LLM 调用点和工具。**

## 四层护栏（按业界"深度防御"标准，逐层加）

### 第 1 层：结构校验（LLM 输出格式对不对）
解决"AI 输出的 JSON 解析失败导致下游崩溃"。

实现：用 Pydantic（Python）或 Zod（TS）给每个 LLM 输出定义 schema：
- 定义期望的字段、类型、取值范围
- LLM 输出后立即校验
- 校验失败 → 带着错误信息自动重试，最多 3 次
- 3 次仍失败 → 明确报错（不要让坏数据流到下游）

参考骨架（Python）见 guardrails/schema_validation.py

### 第 2 层：业务规则校验（内容合不合理）
解决"格式对但内容违反业务规则"（如金额为负、日期倒流、引用不存在的 ID）。

实现：在 schema 里加 validator，把业务规则编码进去：
- 数值范围（价格 > 0、百分比 0-100）
- 引用完整性（引用的 ID 必须真实存在）
- 业务逻辑（结束日期晚于开始日期）
- 关键操作的置信度阈值（低于阈值不执行，转人工）

### 第 3 层：用户确认门（危险操作执行前拦截）
解决"AI 自作主张做了不可逆的事"。

实现：把操作分级，危险操作必须人确认才执行：
- 🟢 安全操作（读数据、查询）→ 直接执行
- 🟡 写操作（改数据库、发消息）→ 记录日志，可配置是否需确认
- 🔴 危险操作（删数据、付款、改权限、发邮件给真实用户）
  → 必须返回"待确认"状态，等人明确批准才执行，绝不自动做

参考骨架见 guardrails/approval_gate.py

### 第 4 层：输入校验（进来的东西可不可信）
解决"恶意输入 / prompt 注入"。

实现：
- 用户输入、网页内容、文件内容都当"不可信"
- 不直接把外部内容当指令执行（防 prompt 注入）
- 长度限制、格式校验
- 涉及 PII 的做脱敏

## 把护栏纳入功能清单

为每个加的护栏，在 feature_list.json 加一个 category 为 "guardrail" 的条目，
让它走正常的 /work + /verify 流程，确保护栏本身也被测试。

## 生成护栏测试

护栏最怕"装了但不生效"。为每层护栏写测试：
- 故意喂格式错误的数据 → 确认被拦下并重试
- 故意喂违反业务规则的数据 → 确认被拒绝
- 故意触发危险操作 → 确认停下来等确认而不是直接做

## 铁律
- 硬约束必须是代码层面的强制校验，不是提示词里的请求。
- 危险操作（删除/付款/改权限）永远需要人确认，这条不可配置关闭。
- 校验失败要么重试要么明确报错，绝不让坏数据静默流过。
- 每层护栏都要有对应测试，证明它真的会拦。
- 📍 下一步：护栏加完，用 /verify 验证它们真的生效，再 /audit 检查覆盖。
