# 共享规则（所有命令通用，集中管理便于升级）
# 这个文件被各命令引用，避免规则散落在 15 个文件里改不全

## 通用启动规则（凡是要动文件的命令，开头都遵守）

### 目录锁定
运行 pwd 报告完整路径给用户。除非用户明确指定，否则在当前目录工作，
绝对禁止 cd 到其他目录、禁止因为别处有同名项目就切换过去。
当前目录不像项目目录就停下问用户。

### 进度衔接
开始实质工作前，先读 claude-progress.txt 最后 20 行 + feature_list.json，
了解上次做到哪、当前 passing/failing 数量。这是"换会话不丢进度"的机制。

## feature_list.json 规则（唯一进度真相源）

- 只允许修改 passes 字段。
- 唯一例外：/change 和 /pivot 命令可以追加新条目、标记 deprecated，
  但必须经用户逐条确认。
- 删除、改写、合并功能描述是不可接受的——会导致功能缺失或带病通过。
- passes 翻 true 的唯一条件：通过端到端验证（不是单元测试通过，
  不是代码写完，是像真实用户一样完整走一遍确认好使）。

## 验证规则（防止"假装做完"）

- 后端/API 功能：用真实输入跑入口命令或发请求，确认输出正确。
- UI 功能：用浏览器自动化截图，或让用户在 localhost 人眼确认。
  UI 功能的最终验收必须有人眼或自动化测试参与，不能 AI 自说自话。

## Git 规则

- 提交格式：<类型>(<范围>): <描述>。类型见 CLAUDE.md。
- 每完成一个功能提交一次。
- 禁止 git push、禁止删分支、禁止操作远程——这些由人类执行。
- 回退用 git reset --hard <commit>，但回退前必须告诉用户会丢失什么、等确认。

## 安全红线（任何命令都不得突破）

- .env 文件永远不进 git。生成 .gitignore 时必须包含 .env。
- 不把 API Key、密码、token 写死在代码里——用环境变量。
- 不在日志、错误信息里打印敏感信息。
- 遇到需要真实凭证、真实付费、删除数据的操作，停下来等用户。

## 会话纪律

- 一个会话只做一个功能（/auto 用子代理隔离除外）。
- 卡住超过 3 次同类尝试 → 停下，写进 progress，建议 /checkpoint。
- 会话结束必须更新 claude-progress.txt。
- 每个命令结束输出"📍 下一步"，告诉用户接下来做什么。
